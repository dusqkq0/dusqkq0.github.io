PROJECT 02

선물하기 시스템

B2C → B2B 다수 수신 구조 재설계 · NHN Commerce 고도몰

2025.7 ~ 2026.3 NHNCommerce

핵심 설계 포인트

왜 기존 구조로는 안 됐나

기존 선물하기는 "구매자 = 수신자"를 전제한 단일 주문 구조. 기업 고객이 직원 50명에게 선물을 보내려면 50번 개별 주문해야 하는 운영 비효율이 VOC로 반복 접수됨.

설계 방향

선물을 "상태를 가진 독립 엔티티"로 재정의. 구매 주체(Buyer)와 수신 주체(Recipient)를 분리하여 B2C 단건부터 B2B 다수 수신(최대 50명)까지 단일 플로우에서 처리 가능하도록 설계.

MVP 범위 결정 이유

수신인 50명 상한선은 무제한 확장이 아닌, CS·정산·취소 리스크를 MVP 기준으로 관리 가능한 범위로 설정한 것. 이후 정책 확장만으로 스케일 가능하도록 구조 설계.

핵심 의사결정 과정

🚧 도전 과제

LA 본사에서는 VIP 등급 기준을 "연 주문 건수"로 잡길 원했고, 한국팀은 "연 매출액" 기준을 주장했습니다. 패션 B2B 특성상 대형 벤더는 단가가 높고 소량 주문하는 반면, 소형 벤더는 다수 소액 주문을 넣기 때문에 건수 기준으로는 실제 고가치 고객을 제대로 분류하지 못하는 문제가 있었습니다.

💡 의사결정

2022~2024년 Heap 데이터를 분석해 "매출 $300K 이상 구매자"가 전체 거래액의 약 62%를 차지함을 수치로 제시했습니다. 이를 LA 팀에 영어 분석 보고서로 공유하여 "연 매출액" 기준으로 정책을 최종 합의했습니다.

✅ 결과

VIP 등급 기준이 명확해지면서 관리자 운영 공수가 줄고, 판매자들의 "왜 나는 VIP가 아닌가" 문의가 출시 후 1분기 내 70% 감소했습니다.

통합예약 UX 개선

2021.6 ~ 2022.1 야놀자

프로젝트 배경

역할

  • UX/UI 기획 및 UT(사용성 테스트) 설계·운영
  • PDP, 통합예약, 로그인/회원가입 플로우 기획
  • 경쟁사 리서치 (에어비앤비, 부킹닷컴, 호텔스닷컴)

문제 정의

  • PDP 진입 후 예약 전환율 XX% (업계 평균 대비 낮은 수준)
  • 사용성 테스트 결과 "날짜 선택 → 인원 설정" 구간에서 이탈 집중
  • 숙박 유형별(호텔/모텔/펜션) 정보 구조가 달라 사용자 혼란

리서치 단계

PROJECT 06

캐시워크 — 리워드 참여 구조 설계

MAU 500만 달성한 만보기 리워드 서비스

2016.9 ~ 2019.5 캐시워크

내가 만든 것 (본인 기여 명확화)

  • 만보기 리워드 적립 구조 설계 (걸음 수 → 캐시 전환 알고리즘 정책 수립)
  • 잠금화면 → 뉴스피드 → 리워드 상품 연결 사용자 플로우 설계
  • 친구 초대 바이럴 루프 기획 (당첨 확률 증가 메커니즘)
  • 광고주 대상 관리자 운영 화면 기획
  • UX/UI 디자인 직접 수행 (초기 스타트업 구조)

핵심 설계 의사결정

도전

초기 리워드 단가를 너무 높게 설정하면 광고 수익보다 포인트 지급 비용이 커지는 구조적 문제 발생 가능성.

결정

유사 서비스 3개(미국 StepBet, 일본 토보쿠, 국내 와이프) 수익 구조 분석 후, 걸음 수 구간별 차등 단가 정책을 설계. 1~5,000걸음 구간 단가를 5,001~10,000 구간보다 낮게 설정해 DAU 체류시간과 광고 노출량을 동시에 최적화.
